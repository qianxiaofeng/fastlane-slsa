# 信任模型与安全假设

[`PROVENANCE.md`](PROVENANCE.md) 拆解了证据链"怎么环环相扣"。本文回答一个更根本、也更
容易被回避的问题：

> 这条链最终把信任落在**谁**身上？SLSA 的安全前提，是不是建立在"GitHub runner 环境是
> 安全的"这个假设上？

**简短回答：是的——而且不止 runner，是整个"构建 + 签发基础设施"。这正是理解 SLSA 的关键，
也是它最常被误解的地方。**

---

## 1. 核心命题：SLSA 不消除信任，而是转移并收敛信任

没有这套机制时，你要相信一个 `.ipa`，实际上**隐含地**要相信一长串东西：打包它的那台
开发者笔记本没中招、那个人没在本地偷改源码、上传渠道没被掉包、CI 脚本没被人塞私货……
这种信任是**分散的、不可见的、人为的**。

SLSA 做的事，是把这一长串塌缩成一句**可机器验证的断言**："它来自这个公开仓库的这条
公开流程"。代价是：你现在必须**信任那个替你盖章的平台**。

> **信任没有消失，它被搬到了一个集中、公开、被许多方同时监督的地方。**

所以"SLSA 让产物无需信任"是错误理解。准确的说法是：它把"你需要信任什么"从无解，
变成一个**有限、显式、可审计、且无权限者无法伪造**的清单。

---

## 2. 信任根：这条链假设了什么是可信的

整条链能成立，等价于**同时假设以下都没有被攻破**：

| # | 假设 | 一旦不成立的后果 |
|---|------|------------------|
| 1 | **GitHub 的 OIDC 签发器是诚实的**——给 token 填的 `job_workflow_ref` 真实反映正在跑的 workflow，不张冠李戴 | 整条链不可伪造性的源头崩塌：可给任意构建签发错误身份 |
| 2 | **GitHub runner 真隔离、真一次性**，且构建任务**无法窃取或伪造 OIDC token 的签发过程**（token 由 GitHub 注入，不是构建脚本能凭空造的） | 构建脚本可导出/伪造签名材料，provenance 失去意义 |
| 3 | **runner 镜像本身干净**（macOS 镜像没被上游或平台植入后门） | 产物在"可信流程"内被悄悄替换，验证仍通过 |
| 4 | **Sigstore/Fulcio 正确校验 OIDC token 才签证书；Rekor 透明日志的 append-only 完整性成立** | 签名链或透明日志可被伪造/篡改 |
| 5 | **写权限边界可信**：能 push `v*` tag、能改 `build-sign-attest.yml` 的人是可信的 | 有权限者可让 workflow 构建任意东西并拿到**完全合法**的签名 |

前 4 条是**平台与基础设施假设**（GitHub + Sigstore + Rekor），靠密码学与平台工程保证；
第 5 条是**人治假设**，密码学保护不到，只能靠权限管理兜底。

---

## 3. SLSA 明确**不**防什么（诚实地说边界）

- **构建平台本身被攻陷**：若 GitHub 内部被渗透、OIDC 签发器被攻破、runner 镜像被投毒——
  provenance 可被**合法地**伪造，`gh attestation verify` 照样退出码 0。SLSA 对此**不设防**，
  这超出它的威胁模型，术语叫"假设一个 **trusted build platform**"。
- **有权限的内鬼**：谁能修改那个 reusable workflow，谁就能让它构建任意东西并拿到完全合法
  的签名。SLSA 防的是"**没有权限的外部篡改**"，**不防**"**有权限的人作恶**"（对应第 5 条假设）。
- **Apple 侧凭证泄漏**：`MATCH_PASSWORD`、App 私钥、app-specific password 的保密性是另一套
  问题（影响代码签名与上传），与 provenance 的可信度正交——见 [`ARCHITECTURE.md` 第 4 节](ARCHITECTURE.md)。

---

## 4. 对本项目的具体落点

- **GitHub-hosted runner 让假设 2、3 比较站得住**：本项目用 `macos-26` 托管 runner，它确实是
  ephemeral、隔离的。**如果换成 self-hosted runner**，隔离性与一次性得自己保证，假设 2 会
  脆弱得多——这也是 SLSA L3 对 self-hosted 更挑剔的原因。
- **第 5 条（写权限）是本项目最现实的薄弱环**：保护它的手段不在密码学，而在工程治理——
  分支保护、tag 保护规则、reusable workflow 抽到受保护的中心仓库（见 [`ARCHITECTURE.md` 第 7 节](ARCHITECTURE.md)）、
  required reviews。把 reusable workflow 集中管控、调用方改不了它，正是为了缩小这一条的暴露面。
- **签名不依赖任何长期密钥**：provenance 的签发用的是 GitHub 运行时现签的短期 OIDC token，
  不是仓库 secret，构建脚本无法导出（见 [`PROVENANCE.md` 第 6 节](PROVENANCE.md)）。这让"密钥泄漏导致伪造"
  这条攻击路径在溯源链上**根本不存在**。

---

## 5. 那 SLSA 的价值在哪？既然终究要信任平台

价值在于**信任的"形状"变好了**：

| 维度 | 没有 SLSA | 有 SLSA |
|------|-----------|---------|
| **可见性** | 隐式——你不知道自己在信任谁 | 显式——白纸黑字：信任 GitHub + Sigstore + Rekor，就这几家 |
| **面积** | 分散——每台开发机、每个能改 CI 的人 | 集中——收敛成一小撮基础设施 |
| **可审计性** | 私密、事后无据可查 | 公开——Rekor 是公开日志，签发记录人人可查 |
| **攻击者性价比** | 便宜——黑一台开发机 / 掉包一次上传即可伪造来源 | 昂贵——必须攻破 GitHub 的签发基础设施，且会在透明日志留痕 |

**一句话**：SLSA 的地基确实是"构建 + 签发平台可信"这个假设。它的意义不是"无需信任任何人"，
而是把原本散落在无数环节、看不见摸不着的信任，**收敛成一个最小、公开、被持续监督、且
无权限者无法伪造的信任根**——而这个清单的最后一条（谁能改 workflow / push tag），恰恰是
密码学保护不到、只能靠权限管理兜底的那一环。
